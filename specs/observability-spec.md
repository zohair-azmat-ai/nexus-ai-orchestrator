# Nexus AI — Observability Specification

## Goals

Every request through Nexus AI should be fully traceable from client to response. Observability is not an afterthought — correlation IDs, structured logs, and event records are first-class concerns built into Phase 1.

---

## Correlation IDs

- **Every** incoming HTTP request gets a UUID4 correlation ID
- If the client sends `X-Correlation-ID` header, it is used (propagation)
- If not, a new ID is generated by the middleware
- The ID is set in `contextvars` and available globally in the request lifecycle
- All log records include `correlation_id` automatically via `ContextFilter`
- The ID is returned in the response `X-Correlation-ID` header

---

## Structured Logging

| Field | Description |
|---|---|
| `timestamp` | ISO 8601 UTC |
| `level` | INFO / DEBUG / WARNING / ERROR |
| `name` | Logger module name |
| `message` | Log message |
| `correlation_id` | Request correlation ID (injected by filter) |
| `*` | Additional fields per log call via `extra={}` |

Format: JSON (production), plain text (development toggle)
Library: `python-json-logger`

---

## Event Types

Events are named constants (see `app/services/events/types.py`):

| Event | When |
|---|---|
| `chat.received` | Chat request arrives at route |
| `chat.completed` | Chat pipeline completes successfully |
| `chat.failed` | Chat pipeline raises exception |
| `orchestrator.*` | Each pipeline stage start/complete |
| `agent.selected` | Triage selects an agent |
| `agent.responded` | Agent generates answer |
| `memory.loaded` | Memory stage reads history |
| `retrieval.searched` | Retrieval stage runs search |
| `ingest.*` | Document ingestion lifecycle |
| `system.startup` | App startup |
| `system.shutdown` | App shutdown |

## Metrics (Phase 1: In-Memory)

Tracked via `analytics/aggregator.py`:
- Request count per endpoint
- Agent selection frequency
- Retrieval hit/miss rate
- Error rate
